# Requirements Document

## Introduction

A GitHub Pages hosted website for the AWS Icons for PlantUML project that allows users to browse, search, and filter the full catalog of AWS service icons. The site is generated by a custom Python script using Jinja2 templates, styled with CloudScape Design System visual patterns, and deployed via GitHub Actions. It reads icon data from `scripts/config.yml`, supports switching between published version tags, and includes rendered example PlantUML diagrams.

## Glossary

- **Site_Generator**: A Python script that reads `config.yml` and Jinja2 templates to produce static HTML, CSS, and JavaScript files for the icon browser website.
- **Icon_Browser**: The main page of the generated website where users view, search, and filter AWS icons.
- **Config_Source**: The `scripts/config.yml` file containing the curated mapping of all AWS icons organized by category with Target names and color assignments.
- **Version_Selector**: A UI control that allows users to choose which release version's icons to display.
- **Category_Filter**: A UI control that allows users to filter the displayed icons by AWS service category.
- **Search_Bar**: A text input field that dynamically filters the visible icon list using client-side DOM manipulation.
- **Icon_Card**: A visual element displaying a single icon's PNG image, Target name, category, and PUML include path.
- **Example_Gallery**: A section of the website that lists and renders PlantUML example diagrams from the repository.
- **PlantUML_Proxy**: The public PlantUML rendering service at `http://www.plantuml.com/plantuml/proxy` used to render `.puml` files as images via URL.
- **CloudScape_Styling**: CSS styles and design tokens derived from the AWS CloudScape Design System, applied to the static site for consistent AWS-branded visual appearance.
- **GitHub_Actions_Workflow**: A CI/CD workflow file that builds the static site and deploys it to GitHub Pages on push to the main branch or on release tag creation.
- **Version_Tag**: A git tag in the format `vN.0` (e.g., v1.0, v23.0) representing a published release of the icon set.
- **Icon_Data_JSON**: A JSON file generated by the Site_Generator containing the full icon catalog extracted from Config_Source, embedded in or served alongside the HTML for client-side filtering.
- **Category_Mapping_Table**: A JSON data structure embedded in the generated site that maps category renames, merges, and deletions across Version_Tags, derived from the BREAKING_CHANGES dictionary in `scripts/upgrade.py`.
- **BREAKING_CHANGES**: A dictionary in `scripts/upgrade.py` that documents category renames (RENAMED), icon replacements (REPLACED), icon removals (REMOVED), and icon moves (MOVED) for each version transition.

## Requirements

### Requirement 1: Static Site Generation from Config Data

**User Story:** As a maintainer, I want the icon browser website to be generated from the existing `config.yml` data, so that the site stays in sync with the icon catalog without manual content updates.

#### Acceptance Criteria

1. THE Site_Generator SHALL read category names, icon Target names, icon Target2 names, and color assignments from the Config_Source file.
2. THE Site_Generator SHALL produce a self-contained static website consisting of HTML, CSS, and JavaScript files in a designated output directory.
3. THE Site_Generator SHALL use Jinja2 templates to render the HTML pages.
4. THE Site_Generator SHALL generate an Icon_Data_JSON structure containing all categories, their associated icons, and color mappings for use by client-side JavaScript.
5. WHEN the Site_Generator is executed, THE Site_Generator SHALL complete without errors when the Config_Source file is valid YAML conforming to the existing schema.
6. THE Site_Generator SHALL be executable via `uv run` from the project root or scripts directory, consistent with the existing build tooling.

### Requirement 2: Icon Browsing and Display

**User Story:** As a user, I want to see all AWS icons organized by category with their names and images, so that I can find the icon I need for my PlantUML diagram.

#### Acceptance Criteria

1. THE Icon_Browser SHALL display all icons from the Config_Source grouped under their respective category headings.
2. THE Icon_Browser SHALL display each icon as an Icon_Card containing the PNG image, the Target name, and the PUML include path (e.g., `Analytics/Athena.puml`).
3. THE Icon_Browser SHALL load icon PNG images from the GitHub raw content URL pattern `https://raw.githubusercontent.com/awslabs/aws-icons-for-plantuml/{version}/dist/{Category}/{Target}.png` using the currently selected version.
4. THE Icon_Browser SHALL apply the category color from Config_Source as a visual accent on each category section header.
5. THE Icon_Browser SHALL display the total count of icons and the count of currently visible icons.

### Requirement 3: Version Switching

**User Story:** As a user, I want to switch between published release versions of the icon set, so that I can view icons as they appeared in a specific release.

#### Acceptance Criteria

1. THE Version_Selector SHALL present a list of available Version_Tags: v13.0, v13.1, v14.0, v15.0, v16.0, v17.0, v18.0, v19.0, v20.0, v22.0, v23.0.
2. THE Version_Selector SHALL default to the latest version (v23.0) on initial page load.
3. WHEN a user selects a different version from the Version_Selector, THE Icon_Browser SHALL update all icon image URLs to reference the selected Version_Tag.
4. WHEN a user selects a different version, THE Icon_Browser SHALL update the displayed PUML include paths to reference the selected Version_Tag.
5. THE Version_Selector SHALL preserve the user's version selection when category filters or search terms are applied.

### Requirement 4: Category Filtering

**User Story:** As a user, I want to filter icons by AWS service category, so that I can narrow down the icon list to the category I am working with.

#### Acceptance Criteria

1. THE Category_Filter SHALL list all categories present in the Config_Source for the currently selected version, as the set of available categories varies across Version_Tags due to renames, merges, and deletions.
2. WHEN no category filter is selected, THE Icon_Browser SHALL display icons from all categories.
3. WHEN a user selects one or more categories from the Category_Filter, THE Icon_Browser SHALL display only icons belonging to the selected categories.
4. THE Category_Filter SHALL toggle icon visibility using client-side DOM manipulation without page reloads.
5. WHEN a category filter is active, THE Icon_Browser SHALL update the visible icon count to reflect the filtered results.
6. WHEN the selected version changes, THE Category_Filter SHALL update its list of categories to reflect only the categories that exist in the newly selected version.

### Requirement 5: Search

**User Story:** As a user, I want to search for icons by name, so that I can quickly find a specific AWS service icon.

#### Acceptance Criteria

1. THE Search_Bar SHALL accept free-text input from the user.
2. WHEN a user types in the Search_Bar, THE Icon_Browser SHALL dynamically show or hide Icon_Cards by matching the search text against the icon Target name and Target2 name.
3. THE Search_Bar SHALL perform case-insensitive matching.
4. THE Search_Bar SHALL filter icons using client-side DOM visibility toggling without server-side requests.
5. WHEN both a search term and a category filter are active, THE Icon_Browser SHALL display only icons matching both the search term and the selected categories.
6. WHEN the search text is cleared, THE Icon_Browser SHALL restore visibility of all icons that match the current category filter.

### Requirement 6: Icon Detail and Copy Support

**User Story:** As a user, I want to easily copy the PlantUML include statement for an icon, so that I can use it in my diagrams without manually constructing the path.

#### Acceptance Criteria

1. THE Icon_Card SHALL display the full PUML include URL for the icon in the format `!include https://raw.githubusercontent.com/awslabs/aws-icons-for-plantuml/{version}/dist/{Category}/{Target}.puml`.
2. WHEN a user clicks a copy button on an Icon_Card, THE Icon_Browser SHALL copy the PUML include URL to the user's clipboard.
3. WHEN the clipboard copy operation succeeds, THE Icon_Browser SHALL display a brief visual confirmation to the user.
4. IF the clipboard API is unavailable, THEN THE Icon_Browser SHALL select the include URL text for manual copying.

### Requirement 7: Example Diagrams Gallery

**User Story:** As a user, I want to see example PlantUML diagrams rendered on the website, so that I can understand how to use the icons in real diagrams.

#### Acceptance Criteria

1. THE Example_Gallery SHALL list example diagrams sourced from the `examples/` directory and README.md inline examples.
2. THE Example_Gallery SHALL render each example diagram as an image using the PlantUML_Proxy service with the URL pattern `http://www.plantuml.com/plantuml/proxy?idx=0&src={encoded_puml_url}`.
3. THE Example_Gallery SHALL display a title and brief description for each example diagram.
4. THE Example_Gallery SHALL include at minimum the following examples: HelloWorld, Basic Usage, Raw Image Usage, Sequence diagrams, Groups - VPC, and Groups - CodePipeline.
5. WHEN an example diagram fails to render via the PlantUML_Proxy, THE Example_Gallery SHALL display a placeholder message indicating the diagram is unavailable and provide a link to the source `.puml` file.

### Requirement 8: CloudScape Visual Styling

**User Story:** As a maintainer, I want the website to follow the CloudScape Design System visual style, so that the site has a professional AWS-branded appearance.

#### Acceptance Criteria

1. THE Icon_Browser SHALL use typography, spacing, and color patterns consistent with the CloudScape Design System.
2. THE Icon_Browser SHALL use CloudScape design tokens for primary colors, background colors, border radii, and font families where applicable.
3. THE Icon_Browser SHALL include a top navigation bar with the project title "AWS Icons for PlantUML" and a link back to the GitHub repository.
4. THE Icon_Browser SHALL be responsive and render correctly on viewport widths from 320px to 1920px.
5. THE Icon_Browser SHALL meet WCAG 2.1 Level AA color contrast requirements for all text elements.

### Requirement 9: GitHub Actions Build and Deploy

**User Story:** As a maintainer, I want the website to be automatically built and deployed to GitHub Pages via GitHub Actions, so that the site is always up to date with the latest release.

#### Acceptance Criteria

1. THE GitHub_Actions_Workflow SHALL be defined in `.github/workflows/deploy-pages.yml`.
2. WHEN a push occurs to the main branch, THE GitHub_Actions_Workflow SHALL build the static site using the Site_Generator.
3. WHEN a release tag is created, THE GitHub_Actions_Workflow SHALL build and deploy the static site to GitHub Pages.
4. THE GitHub_Actions_Workflow SHALL use `uv` to install Python dependencies and execute the Site_Generator.
5. THE GitHub_Actions_Workflow SHALL deploy the generated output directory to GitHub Pages using the `actions/deploy-pages` action.
6. IF the Site_Generator fails during the build step, THEN THE GitHub_Actions_Workflow SHALL fail the workflow run and report the error.

### Requirement 10: Accessibility

**User Story:** As a user with assistive technology, I want the icon browser to be accessible, so that I can navigate and use the site effectively.

#### Acceptance Criteria

1. THE Icon_Browser SHALL use semantic HTML elements (nav, main, section, header, footer) for page structure.
2. THE Icon_Browser SHALL provide alt text on all icon images using the icon Target name and category.
3. THE Icon_Browser SHALL ensure all interactive controls (Search_Bar, Category_Filter, Version_Selector, copy buttons) are keyboard-navigable.
4. THE Icon_Browser SHALL use ARIA labels on interactive controls that lack visible text labels.
5. THE Icon_Browser SHALL maintain a logical tab order through the page controls and icon grid.

### Requirement 11: Cross-Version Category Mapping

**User Story:** As a user, I want my active category filter to automatically map to the correct category name when I switch versions, so that I do not lose my filter context due to category renames or deletions across releases.

#### Acceptance Criteria

1. THE Site_Generator SHALL read category rename, merge, and deletion data from the BREAKING_CHANGES dictionary in `scripts/upgrade.py` for all supported Version_Tags (v13.0 through v23.0).
2. THE Site_Generator SHALL generate a Category_Mapping_Table and embed it in the generated site as a JSON data structure accessible to client-side JavaScript.
3. THE Category_Mapping_Table SHALL contain forward mappings for each version transition where a category was renamed: v13.0 (ARVR → VRAR, AWSCostManagement → CloudFinancialManagement), v13.1 (GroupIcons → Groups), v15.0 (GameTech → Games), v19.0 (MachineLearning → ArtificialIntelligence, MigrationTransfer → MigrationModernization).
4. THE Category_Mapping_Table SHALL contain deletion markers for each version transition where a category was removed: v16.0 (VRAR → deleted), v22.0 (Robotics → deleted), v23.0 (ContactCenter → deleted).
5. WHEN a user switches versions while a category filter is active, THE Icon_Browser SHALL resolve the selected category name through the Category_Mapping_Table to determine the equivalent category name in the target version.
6. WHEN the Category_Mapping_Table indicates the selected category was renamed between the source and target versions, THE Icon_Browser SHALL automatically update the active category filter to the renamed category in the target version.
7. WHEN the Category_Mapping_Table indicates the selected category was deleted in the target version or a version between the source and target, THE Icon_Browser SHALL clear the active category filter and display all categories.
8. WHEN a user switches from a higher version to a lower version (e.g., v19.0 to v18.0), THE Icon_Browser SHALL resolve category names in reverse, mapping the newer name back to the older name (e.g., ArtificialIntelligence → MachineLearning).
9. WHEN a user switches versions and the active category exists unchanged in the target version, THE Icon_Browser SHALL preserve the active category filter without modification.
